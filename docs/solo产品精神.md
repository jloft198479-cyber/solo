### solo-markdown 产品精神

#### 产品背景

做这款产品的起因是，市场上有很多 Markdown 编辑器，数不胜数。最有名的应该是 Typora，最近比较火的应该是 Obsidian，总之种类各样都有。我也下载体验了很多，但感觉都不太舒服。

其实像 Obsidian 这样的专业知识库 Markdown 编辑器，我也有在用，但它主要用于做知识库。代码编辑方面，Notepad 做得非常不错。但我本质上想要的，是一个类似 Windows 自带记事本那样轻巧的 Markdown 编辑器。它是单文件形式，我可以在桌面随时创建 Markdown 文件，就地修改，并保存到任何地方。它不需要像知识库那样有侧边栏或复杂功能。

我不希望它承担太多功能，因为工作中我追求的是利用碎片化时间快速表达想法。

基于此，我重新构建了这款产品，**叫 Solo，寓意“心流”**。它可以随时打开、关闭、保存，不会给我带来压力或负担，这是我开发的初衷。

由于这类产品市场很多，没必要重复造轮子，我们在 GitHub 上找了源码，在此基础上进行了大幅度的删减、优化和修改，最终形成了这款产品。只要能达到目的即可，这就是产品的基本背景介绍。

#### 产品的基本要求

我对这个产品的要求，基本沿袭了我一贯的设计理念。我利用 Agent开发了很多产品，核心思想大同小异，主要包含三大原则：

\*\*第一，极简。\*\*极简意味着产品不应赋予过于复杂的功能展现，以免让用户无所适从。其表现必须极简，即“做减法，不做加法”。

\*\*第二，极速。\*\*速度是衡量产品好坏的关键指标。如果打开产品耗时过长，用户体验会非常糟糕。以记事本为例，我们要做到秒开秒关，这种即时快感能带来极佳的体验，让用户将注意力集中在内容上，而非无关的操作上。所以速度至关重要

\*\*第三，优雅。\*\*除了极简和极速，产品还应展现优雅的一面，这也是我们产品带有文艺气息的原因。视觉上必须让人感到舒适、流畅，这样才能帮助用户顺利进入工作流，避免被无关要素打扰。

此外，在框架结构上，我一般检查三个原则：**灵活、高效、可拓展**

关于灵活性，通常指的是在产品开发中实现\*\*“原子化、组件化”设计，遵循高内聚、低耦合的原则。\*\*这样每个模块既能有效协作，又能保持独立，相互之间不会产生过多依赖，同时又能有机组合，比如字体模块，应该按需加载，可以保证启动速度快、使用流畅；而不需要的字体则无需加载。从技术架构来看，渲染部分应作为独立模块。我们常遇到将代码全部塞进一个文件的情况，这会导致排查困难、问题难解，且修改一处易引发连锁反应。此外，代码耦合度高还会导致各种奇怪的问题，使得调整产品结构变得极其困难。因此，我们需要对产品结构进行有效拆分，在保证各模块独立性和灵活性的同时，实现有机的调用。这是我追求的产品基本框架。

有了这种灵活性，系统的可拓展性也就建立起来了。什么是可拓展性？当你想要开发新功能或增加新模块时，这个过程会变得相对容易，通常只需要一个接口或一段代码，就能将这些功能串联起来。否则，如果把代码都堆在一起，增加功能时需要权衡的因素就太多了，反而不利于后期的混合开发。因此，我们的产品非常强调可拓展性。例如，在 Solo 中，我们就集成了微信导出、网页导出等拓展功能，我认为这非常重要。

当然还有高效。什么是高效？这是我一直追求的，即产品必须耗能少。例如，对 CPU 的能耗非常小，对内存的要求也非常低，内存压力很小。因此，我常使用 Rust 这种轻量级技术方案，而非大众常用的网页技术方式，从而为用户提供更好的体验。首先，安装包小，几兆或十几兆即可实现所需功能，无需安装几百兆的产品；其次，内存占用小。现在的 PC 软件动不动就占用几百兆甚至上 G 内存，而我们希望任何产品的内存占用都不超过一百兆或几十兆。这样不仅保证了用户能运行更多任务，还避免了对系统造成过大负担。

简约而不简单，我们追求的是用最少的代码，开发出性能极致的产品，这是一贯坚持的基本原则。

`关键词：极简、极速、优雅、灵活、高效、可拓展`
