# Editor Input Scenarios

更新时间：2026-06-12

这份文档用于梳理 MarkLight 预览态编辑器里的 Markdown 直输、IME 输入、以及“应支持/未支持/疑似有 bug”的场景。

建议测试环境：

- 版本：`v1.1.3`
- 视图：预览/所见即所得编辑模式
- 输入法：先用英文输入法，再切中文拼音输入法
- 文档：空白新文档开始，避免历史内容干扰

## 结论摘要

### A. 理论上应该生效，若无效则是 bug

- `# ` 标题输入
- `**加粗**`
- `__加粗__`
- `*斜体*`
- `_斜体_`
- `~~删除线~~`
- `` `行内代码` ``
- `==高亮==`
- `^上标^`
- `~下标~`
- `![](图片地址)` 图片
- `[[wikilink]]`

### B. 当前看起来本来就不支持 Markdown 直输

- `[文字](https://example.com)` 链接
- `| a | b |` 这类表格直输
- 仅输入单个符号后自动变格式，如单独输入 `!`、`[`、`]`、`(`、`)`

### C. 已知高风险问题

- `# ` 后立刻输入中文，可能出现 `# n你`
- 本该吃掉的 Markdown 标记如果没转成格式，保存/序列化后可能变成转义形式
  - 例：`\*\*加粗\*\*`

## 测试方法

每个场景建议都测两次：

1. 英文输入法
2. 中文拼音输入法

每次记录三件事：

- 是否即时渲染成目标格式
- 是否残留 Markdown 标记字符
- 保存后是否被自动加了转义字符

## 场景清单

### P0: IME / 标题问题

#### 场景 P0-1：`# ` 后输入英文

输入：

```md
# hello
```

步骤：

1. 输入 `#`
2. 输入空格
3. 立刻输入 `hello`

预期：

- 渲染成 H1
- 不残留额外字符

#### 场景 P0-2：`# ` 后输入中文

输入意图：

```md
# 你
```

步骤：

1. 切中文拼音输入法
2. 输入 `#`
3. 输入空格
4. 立刻输入拼音 `ni`
5. 上屏“你”

异常表现：

- `# n你`
- `# ni你`
- 光标错位
- 候选上屏后跑到下一段

这是当前最需要优先确认的场景。

### P0: 基础 inline Markdown 直输

#### 场景 P0-3：加粗

输入：

```md
**加粗**
```

预期：

- 自动转成加粗文本
- 不应保留 `**`
- 保存后不应变成 `\*\*加粗\*\*`

#### 场景 P0-4：加粗（下划线写法）

输入：

```md
__加粗__
```

预期：

- 自动转成加粗文本

#### 场景 P0-5：斜体

输入：

```md
*斜体*
```

以及：

```md
_斜体_
```

预期：

- 自动转成斜体
- 不残留包裹符号

#### 场景 P0-6：删除线

输入：

```md
~~删除线~~
```

预期：

- 自动转成删除线

#### 场景 P0-7：行内代码

输入：

```md
`code`
```

预期：

- 自动转成行内代码样式

#### 场景 P0-8：高亮

输入：

```md
==高亮==
```

预期：

- 自动转成高亮

### P1: 扩展直输

#### 场景 P1-1：上标

输入：

```md
^sup^
```

预期：

- 自动转成上标

#### 场景 P1-2：下标

输入：

```md
~sub~
```

预期：

- 自动转成下标

#### 场景 P1-3：Wikilink

输入：

```md
[[page]]
```

以及：

```md
[[page|alias]]
```

预期：

- 自动转成 wikilink 内联节点

#### 场景 P1-4：图片

输入：

```md
![alt](https://example.com/demo.png)
```

预期：

- 若支持图片直输，应转成图片节点
- 若没生效，记录是否被转义

### P2: 当前大概率未支持，但要确认

#### 场景 P2-1：Markdown 链接

输入：

```md
[OpenAI](https://openai.com)
```

当前预判：

- 大概率不会自动转成链接 mark
- 可能保留原文本，或保存后加转义

#### 场景 P2-2：表格直输

输入：

```md
| A | B |
|---|---|
| 1 | 2 |
```

当前预判：

- 大概率不会自动转成表格节点

### P2: 转义回写检查

这些场景即使输入时没立刻出问题，也要做“保存/切换文档/重新打开”复查。

#### 场景 P2-3：未被识别的 Markdown 标记是否被转义

重点检查：

```md
**加粗**
*斜体*
~~删除线~~
`code`
[link](https://example.com)
| a | b |
```

异常表现：

- 变成：

```md
\*\*加粗\*\*
\*斜体\*
\~\~删除线\~\~
\`code\`
\[link\]\(https://example.com\)
\| a \| b \|
```

## 建议记录格式

可以按下面格式逐条记录：

```md
- 场景：P0-3 加粗
- 输入法：中文拼音
- 是否即时生效：否
- 是否残留符号：是
- 保存后是否被转义：是
- 实际结果：\*\*加粗\*\*
```

## 当前判断

如果你只想先看最关键的，优先测这 4 个：

1. `# ` 后输入中文
2. `**加粗**`
3. `*斜体*`
4. `[文字](https://example.com)`

这 4 个基本就能区分：

- 是 IME/composition 残留问题
- 还是 Markdown 直输规则缺失
- 还是序列化转义问题
