## 1. solo 与「墨色书卷」设计系统（评估语境）

solo 是一个 **Tauri v2 桌面端 Markdown 编辑器**（Vue 3 + TipTap/ProseMirror + Rust），本地优先、无后端、单文件编辑，定位「中文沉浸式写作」。

现有设计系统（已完成、勿重复设计）：
- **配色基调**：暖棕 + 宣纸（`--bg-color:#faf8f5` / `--primary-color:#8b7355`），明暗双主题 + 7 套预设（schola / elegant / cinnabar / default / gray-domain …）。
- **主题机制**：基于 CSS 变量。`applyTheme()` → `injectColors()` 重写 `:root` 变量；切换时加 `theme-transitioning` 类做 300ms 全局过渡防闪。**架构已为侧栏预留专属变量 `--sidebar-bg` / `--sidebar-hover`**（`themes/types.ts` 的 `ThemeColors`）。
- **动效语言**：克制统一，120ms（按钮/菜单）/ 200ms（面板/视图）ease-out（`main.css`）。
- **已打磨的块级元素**：代码块、表格、mermaid、callout、数学公式均已有样式。
- **标题栏**：自绘标题栏，当前右侧按钮区顺序为 `焦点(眼睛) → 置顶(图钉) → 最小化 → 最大化 → 关闭`（`CustomTitlebar.vue:40-69`，注意：模板内视图切换组与窗口控制组之间**无分隔符**）；D3 将在此顺序中插入大纲按钮（见第 2 节）。
- **焦点模式**：隐藏 UI，hover 顶部 12px 触发条才淡入标题栏（`main.css` focus-mode 规则）。

## 2. 已锁定的设计决策（不可动摇，评估时视作约束）

以下结论来自用户与方案的反复沟通，第三方评估须在约束内提方案：

| # | 决策 | 理由（已核实） |
|---|------|----------------|
| D1 | **大纲放右侧，不放左侧** | 左侧在用户心智里是「文档切换区」，solo 不做文档切换故留空；且左缘已有标题 `H1/H2/H3` gutter 徽标（`editor.css` 用 `::before` 负偏移画在左 margin），再压侧栏会左缘拥挤。 |
| D2 | **大纲可自由收缩** | 沉浸式写作时收起以保落笔点干净；用户主动唤出。 |
| D3 | **打开入口 = 顶部标题栏新增常驻大纲按钮 + `Ctrl+/`** | 最终按钮顺序：**`焦点(眼睛) → 大纲(新) → 置顶(图钉) → 最小化 → 最大化 → 关闭`**。理由：① 普通模式标题栏常驻（opacity:1）发现性最高；② 大纲插在焦点与置顶之间，使「焦点+大纲」两个内容视图切换相邻、与「置顶」共同构成视图切换组，语义自洽；③ 视图切换组与窗口控制组（最小化/最大化/关闭）之间建议加一条分隔线，使两组边界清晰（当前 `CustomTitlebar.vue` 模板无分隔符）；④ `Ctrl+/` 组合键不输入字符、与斜杠命令 `/` 无冲突、中国键盘单手可达。（评估注：此位置＝选项 A，为已锁定决策；若评估者倾向「置顶之后、最小化之前」＝选项 B 亦可提，但会拆开「焦点+大纲」的相邻关系。） |
| D4 | **展开用 push（编辑区左移让位），绝不用 overlay 盖住内容** | 导航栏压住正在写的字，对沉浸式写作是硬伤。 |
| D5 | **侧栏自动跟随主题** | 引用 `--sidebar-bg` / `--sidebar-hover` / `--border-color` / `--primary-color` / `--text-secondary`，零额外主题代码即可适配全部预设的明暗。 |

## 3. 优化项详述

每项结构：**问题证据 → 目标 → 方案 → 交互细节 → 技术落地 → 验收标准**。

### P1 · 右侧大纲导航侧边栏 ⭐核心

**问题证据**
- 大纲数据已在算：`extractEditorOutline()`（`Editor/tiptap/editor-metadata.ts:24`）提取标题层级，`MarkdownEditor.vue:263/277/323` 通过 `emit('update',{outline})` 抛出。
- 但 App 只消费了 `wordCount` / `selectionText` 喂给状态栏（`App.vue:284`、`useAppEditorState.ts:20` 仅存类型未渲染），**已有数据无任何 UI 呈现**——纯前端缺口。

**目标**：把已就绪的大纲数据渲染成可跳转的右侧导航，提升长文写作的结构感与可达性。

**方案**
- 新建 `OutlinePanel.vue`（右侧 flex 子项，宽 ~236px，`flex-shrink:0`），`App.vue` 布局改为 `标题栏 / (编辑区 flex:1 + 大纲 flex 子项) / 状态栏`。
- 复用 `extractEditorOutline` 输出的 `EditorOutlineItem[]`（含层级 level、文本、文档内位置）。
- 点击大纲项 → 编辑器 `editor.commands` 跳转到对应标题节点（按 item 携带的 position/doc 坐标）。

**交互细节**（遵守 D1–D5）
- 位置：右侧（D1）；可收缩，收起后编辑区自动占满（D2）。
- 打开/关闭：标题栏在「焦点」与「置顶」按钮之间新增大纲按钮 + `Ctrl+/`（D3，最终顺序见 D3）。
- 展开：push 模式，编辑区 `margin-right` 让出 236px（D4）。
- 样式：背景 `--sidebar-bg`、hover `--sidebar-hover`、激活项右边框 `--primary-color`、分隔线 `--border-color`（D5）。
- 激活项跟随当前滚动位置高亮（scroll-spy）；收缩态下 hover 当前行仍可临时浮出层级徽标（与 P6 联动）。

**技术落地**
- 数据：已有，无需新增提取逻辑。
- 跳转：ProseMirror `editor.view.state.doc` 节点定位，复用现有 `editor.commands`。
- 状态：在 `useAppEditorState` / 新建 `useOutline` composable 管理 `open/side/items`。
- 主题：纯 CSS 变量，无 JS 主题分支。

**验收标准**
- 打开 `.md` 含多级标题，右侧出现大纲；点击跳转准确；`Ctrl+/` 与标题栏按钮均可 toggle；收起后编辑区无遮挡、无布局抖动；切换明暗/预设主题侧栏同步变化；长文档滚动激活项跟随。

### P2 · 状态栏重设计

**问题证据**
- `App.vue:371` `.minimal-statusbar { opacity: 0.55; }`——常驻透明度 0.55，hover 才提亮，关键信息（保存态）默认偏低调。
- 状态栏是「换肤/换字体」快开的唯一常驻入口（`StatusbarQuickActions.vue`），却藏在最低对比区，难发现（评审⑦）。

**目标**：提升状态栏信息可读性，让快开入口更易发现，同时不破坏克制基调。

**方案**
- 基础透明度 0.55 → 提升至 ~0.85–0.9；「未保存」等关键状态用更高对比 / `--primary-color` 强调。
- `StatusbarQuickActions`（主题/字体快开）保持常驻、图标更明确；与 P1 的大纲按钮分工（大纲在标题栏、主题/字体在状态栏）。

**技术落地**：改 `App.vue` 的 `.minimal-statusbar` 透明度 + `StatusbarQuickActions` 视觉权重；沿用现有 `--text-secondary` / `--muted-color`。

**验收标准**：默认态文字清晰可辨；保存/未保存状态一眼可辨；主题/字体快开图标明显且不喧宾夺主。

### P3 · 命令面板 Ctrl+K ⭐核心

**问题证据**
- 完整命令注册表已存在：`commands/registry.ts:45` 导出 `COMMANDS: CommandDefinition[]`，`findCommandByShortcut`（`commands/registry.ts`）用于快捷键分发（`App.vue:24/162`、`useAppDomEvents.ts:50`）。
- 用户只能靠记快捷键或从菜单翻，**缺一个可视化、可搜索的统一入口**（评审⑥）。

**目标**：基于现有 registry 做模糊搜索命令面板，服务键盘流，零架构成本。

**方案**
- 新建 `CommandPalette.vue`：背景虚化/压暗当前编辑器，中间浮起搜索面板。
- 数据源：直接复用 `COMMANDS`（按 `scope` 分组：文件 / 视图 / 插入 …）+ 各命令已带的 `shortcut` 字段。
- 模糊匹配命令 label / 关键字；方向键导航、回车执行；右侧显示快捷键提示；当前高亮项。

**交互细节**
- 触发：`Ctrl+K`（与主流编辑器一致）。
- 浮层用 overlay（此处的 overlay 是**临时模态**，非常驻导航，不违反 D4；D4 仅约束大纲常驻侧栏）。
- 选中即调用现有 dispatcher 执行该 command，复用既有逻辑。

**技术落地**：读 `COMMANDS` + `findCommandByShortcut`；执行路径复用 `useCommandDispatcher`（或等价分发）；无需新增 Rust 命令。

**验收标准**：`Ctrl+K` 唤起；输入关键字实时过滤；分组清晰；快捷键提示准确；回车执行对应命令且面板关闭；空查询展示全量分组列表。

### P4 · 信号色（交互强调色）

**问题证据**
- `--primary-color:#8b7355` 一身兼三职：装饰（标题 gutter、引用线、链接）+ 状态（选中、激活、强调点）。链接、标题标记、激活态全是一个棕，缺第二个对比色区分「可点」与「好看」（评审②）。

**目标**：引入一个克制的互补强调色，专门标「可交互/可点击」，书卷感不破、交互语音更清。

**方案**
- 新增 `--accent-color`（建议墨蓝 `#3b5b7a` 或赭红 `#b5503a` 二选一，**评估阶段须做左右对照小样定夺**，见第 6 节），仅用于：链接 hover、可点击入口、focus ring、当前激活指示。
- `--primary-color` 回归「装饰 + 品牌」语义，不再承担全部交互强调。
- 该色同样进 `ThemeColors`，随主题切换。

**技术落地**：`themes/types.ts` 增 `accentColor` 映射；`main.css` 定义变量；链接/按钮/focus 引用。

**验收标准**：可交互元素与纯装饰元素在色相上可区分；明暗双主题下均可读；不破坏暖棕书卷基调（评估者可对色相/饱和度提异议）。

### P5 · 次级文字对比度达标（WCAG AA）

**问题证据**
- `--muted-color:#9a8e7e` 压在 `--bg-color:#faf8f5` 上实测对比度约 **3.0:1**，低于 WCAG AA 文本要求的 **4.5:1**（设置页描述、占位符、状态栏辅助文字均用此色）。
- 暗色主题 `--muted-color:#6b6258` 同样偏低。

**目标**：把次级文字对比度提到 ≥4.5:1（AA），消除「精致但看不清」的短板。

**方案**
- `--muted-color` 调深一档（建议 `#8a7d6b` 或更深，须实测 ≥4.5:1）；暗色同步调整。
- 与 P2 协同：状态栏关键状态用 `--text-secondary` 或 `--accent-color` 而非 muted。

**技术落地**：仅改 `main.css` 两处变量值 + 验证全部引用处（`--muted-color` 被 40+ 处引用，须回归核对无语义断裂）。

**验收标准**：浅/深主题下 `--muted-color` 对底对比 ≥4.5:1；全文次级文字清晰；不波及 `--text-secondary`（#5c5040，已达标）的层级关系。

### P6 · 标题 gutter 徽标 hover 显现

**问题证据**
- `editor.css` 中 `H1/H2/H3` 徽标常显于左 margin（opacity ~0.45）。短文档好看，长文档左缘持续「贴便利贴」产生视觉噪声（评审④）。

**目标**：默认隐藏层级徽标，仅在 hover 当前行/标题时浮现，长文档更干净。

**方案**
- 徽标默认 `opacity:0`；`:hover` 当前标题块或当前行时 `opacity` 升至 ~0.45。
- 与焦点模式「段落聚焦变暗」同一范式（已是 solo 的好先例）。

**技术落地**：纯 CSS（ProseMirror 节点 hover），无 JS。

**验收标准**：长文档左缘默认干净；hover 标题行才显层级；不干扰 P1 大纲的滚动高亮。

### P7 · 焦点模式退出引导

**问题证据**
- 进入焦点模式后标题栏隐藏，退出靠 hover 顶部浮现眼睛图标；首次用户可能不知如何退出（评审⑧）。

**目标**：给一次性轻提示，降低首次使用困惑。

**方案**
- 进入焦点模式时，顶部细条显示极轻提示「移到顶部退出聚焦」（3–4s 淡出）。
- 顺带收益：因大纲按钮现已与焦点按钮并列（D3），hover 顶部同时暴露「退出聚焦 + 开大纲」两个入口。

**技术落地**：复用 `settingsStore.isFocusMode` 布尔状态作为触发条件；但提示本身**需新增 DOM + CSS**——
- 新增顶部提示元素（可置于 `titlebar-trigger` 区域或独立 `.focus-enter-hint` 节点），含淡入淡出样式（3–4s 后 `opacity:0`）；
- 在焦点模式监听中**新增 enter 分支**（现有 `App.vue:73-85` 的 `watch` 仅处理 `!active` 即「退出」提示，无 enter 分支）；
- 注意：现有 `focusModeNotice`（App.vue:70-85）仅显示「已退出焦点模式」且位于状态栏（App.vue:281），而焦点模式下状态栏被 `opacity-0 pointer-events-none` 隐藏（App.vue:276 + main.css:347），**其 UI 不可直接复用**，本提示是独立于状态栏的新元素。

**验收标准**：首次进入可见提示且不打扰；之后不再出现；hover 顶部能退出并见大纲入口。

### P8 · 斜杠命令首启提示

**问题证据**
- `/` 斜杠命令是 solo 核心特色（已为中文修复 `allowedPrefixes`，见 AGENTS.md 历史经验），但空文档只给占位符，用户需自己知道打 `/`（评审⑩）。

**目标**：空文档给一句轻提示，降低新用户上手门槛。

**方案**
- 空文档占位符追加一行：「输入 `/` 唤起命令」。

**技术落地**：改 `MarkdownEditor.vue` 占位符模板（一处）。

**验收标准**：空文档可见提示；输入 `/` 后提示自然消失；不打扰有内容的文档。

---

### P9 ·（可选）最近文件快开

**问题证据**
- solo 每篇 `.md` 开一个 OS 窗口，无应用内多文档/tab/最近文件切换（类似 Typora 的刻意选择）。切换文档效率低（评审⑨）。

**目标**：在不改变多窗口模型前提下，提供「最近文件」快开，提升切换效率。

**方案**（可选，非 must）
- 命令面板（P3）内增「最近文件」分组，数据源来自现有打开历史 / 文件系统最近记录。
- 或顶栏加一个轻量「最近」入口。

**技术落地**：依赖 Rust 侧是否已有最近文件记录；若无可从 `invoke` 现有窗口/文档 API 取。评估阶段确认可行性。

**验收标准**：可快速在历史文档间跳转；不引入应用内 tab 模型（保持 solo 多窗口哲学）。

## 4. 优先级与落地顺序建议

| 层级 | 项 | 理由 |
|------|-----|------|
| **高杠杆（数据/架构已就绪，纯体验缺口）** | P1 大纲、P3 命令面板 | 不写新内核逻辑，复用 `extractEditorOutline` 与 `COMMANDS`，投入小收益大。 |
| **全局质感** | P4 信号色、P5 对比度 | 改少量 CSS 变量，影响全应用观感与可读性。 |
| **打磨项** | P2 状态栏、P6 gutter、P7 焦点引导、P8 斜杠提示 | 单点改动，风险低，提升完成度。 |
| **可选** | P9 最近文件 | 取决于 Rust 侧数据可用性，非必须。 |

**建议落地节奏**：P1 + P3 先行（核心体验），P4 + P5 同步（质感），其余打磨项跟随。

## 6. 风险与注意点

- **P1 与现有布局的耦合**：`App.vue` 当前为 `标题栏 / 编辑区 / 状态栏` 纵向结构，加右侧大纲需把中段改为横向 flex；须确保编辑器 `overflow`、滚动、焦点模式的布局切换不冲突。
- **P3 overlay 与 D4 不冲突**：D4 仅约束大纲常驻侧栏不得 overlay；命令面板是临时模态，允许 overlay（已说明）。
- **P5 回归面大**：`--muted-color` 被 40+ 处引用，改值后须全量核对语义（如占位符、禁用态、辅助说明）未被误伤。
- **P4 色相选择（需评估阶段定，建议出对照小样）**：墨蓝 `#3b5b7a`（冷、收敛、偏「工具感」）与赭红 `#b5503a`（暖、活跃、偏「书卷感」）色相相差约 180°，对整体气质影响相反。建议评估阶段用同一 UI 片段（链接 hover、focus ring、大纲激活项、命令面板高亮）渲染两色做并排对比，由评估者直接选，避免把决策推到落地阶段反复。本项属设计主观项，非硬伤。
