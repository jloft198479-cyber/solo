# 产品开发总原则

> 本文档定义代码和系统设计的基本原则。专注技术决策。

---

## 核心原则

### 真理源自一处，全局引用

- 配置、状态、常量只有一份，其他地方只引用
- 状态管理中心唯一，通过 `setState()` → 事件通知 → 组件响应，单向数据流

### 高内聚、低耦合

- 一个文件只做一件事
- 组件不直接操作其他组件的 DOM
- 控制器不直接知道视图的细节，视图不直接调 API

### 模块化、组件化、原子化

- UI 组件可独立渲染、独立销毁（有 `init/destroy`）
- 工具函数是纯函数，不依赖全局状态
- 功能单元可单独测试

### 更轻、更快、更强

- 不引入不必要的依赖
- DOM 操作节流（requestAnimationFrame）
- 错误边界保护，一处崩溃不影响全局

### 重结构，轻代码

- 好的结构比多的代码重要十倍
- 先搭架子，再填内容
- 拆分优于堆砌

### 资源节约原则

- **产品设计要替用户节约资源**——省内存、省 CPU、省磁盘
- 功能完成 ≠ 任务完成，资源消耗也是质量指标
- 堆砌代码 = 浪费用户资源 = 工作不合格
- 重复逻辑必须合并，双重处理必须消除
- 代码简洁不是风格偏好，是对用户资源的尊重

#### 决策阶梯（写代码前必须过）

在写任何代码之前，从第一级开始，**停在第一个成立的层级**：

1. **这东西需要存在吗？** —— 推测性需求 = 不做（YAGNI）
2. **标准库/运行时能做吗？** —— 用标准库，不自己造轮子
3. **平台原生功能能覆盖吗？** —— 用原生（CSS 能做的不写 JS）
4. **已安装的依赖能解决吗？** —— 用现有依赖，不为几行代码引入新依赖
5. **能一行搞定吗？** —— 一行
6. **以上都不行** —— 写最少能工作的代码

阶梯是反射动作，不是研究课题。两级都成立 → 取更高级，继续前进。

### 绝不硬编码

- 任何数据、配置、路径都不允许写死在代码中
- 所有动态内容必须从单一数据源派生
- 硬编码 = 绕过"真理源自一处"原则 = 数据不一致

### 契约设计

- 模块间交互必须通过明确契约（接口签名、事件格式、数据结构），不允许隐式依赖
- 契约一旦确定，消费者只依赖契约，不依赖实现细节

### 退化安全

- 任何外部依赖都可能失败，**必须有 fallback**
- 降级策略必须提前设计
- 用户永远不应看到空白页面或无响应——至少给出明确的错误提示

#### 绝不简化掉的清单

- 信任边界的输入校验（用户输入、外部数据）
- 防止数据丢失的错误处理
- 安全措施（路径遍历防护、XSS 防护）
- 用户明确要求保留的功能

### 简化标记约定

当按决策阶梯做了刻意简化时，用 `simp:` 注释标记：

```js
// simp: 全局锁，单账户吞吐量不足时改为 per-account 锁
```

```css
/* simp: 硬编码颜色，主题化时提取为 CSS 变量 */
```

- 格式：`simp: <已知上限>, <升级触发条件>`
- 简单的一行代码无需标记
