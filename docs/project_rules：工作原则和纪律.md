# 项目工作原则

> 本文档是项目所有工作的最高准则。任何 AI 助手、任何开发者、任何会话，必须遵守以下原则。 违反其中任何一条，都视为工作不合格。
> 
> \*\*每次会话必须同时读取 \*\*`产品需求说明.txt`，了解产品基本情况，避免重复解释。

---

## 一、工作原则

### 实事求是

- 充分调研，以事实为基础
- **不瞎蒙，不猜测**——不确定的事情必须验证后再说
- 读源码、读文档、读实际文件，不准凭空推断
- 处理代码问题必须读源码、源文档，不依赖第三方描述就下判断

### 第一性原理，必要时回到扎实的地基重建

- **怀疑**：觉得「大家都这么干」的前提不对，就值得重新思考
- **动手**：回到底层，能用更根本的理由去掉一个默认值又不坏事，它本来就不该有
- **节制**：普通问题顺藤摸瓜查原因就行，别为显摆硬重推

### 最佳实践

- 务必结合成功实践，绝不闭门造车
- 参考成熟开源项目的优秀架构
- 卡住时：死循环 / 长期无进展必须立即停止，去查最佳成功实践或向外求援，而非硬扛

### 策略大于行动

- **最佳策略远胜于反复琢磨细节**
- 先想清楚整体方案，再动手。方向对了，细节才会对
- 想透再执行，不急着写代码
- 好策略更省时省力、少耗 token，结果反而更好

### 结构性看问题

- 结构性思维才能换来极简操作
- 先看全局依赖关系，再动局部代码
- 举一反三：一个问题常代表一类问题，上升一个维度，别在细枝末节绕弯子

### 消除理解偏差，取得共识再行动

- 任何改动前必须与用户沟通确认
- **绝不允许"用户一说完就噼里啪啦敲代码"**
- 动手前必须复述理解，确认与用户意图一致——**理解偏差是最大的浪费**
- 术语和概念必须精确对齐——"直接对话"和"委托执行"是完全不同的模式，不能混用
- 遇到模糊指令，先问清含义再行动，宁可多问一句，不可猜错一步
- 读代码时以实际行为为准，不以注释或文档为准——**代码是唯一的真相**
- 修改前先确认影响范围：改了这个文件，还有哪些文件会受影响？逐个检查

### 分步行动，渐进实现

- 不要一次性完成所有工作
- 每阶段完成后停下来，确认无误再继续
- 每步都要复查验证
- 先跑通最小可用版本，再逐步增强——**能跑 \> 完美**
- 每一步必须可独立验证——能跑、能用、能测
- 不为假设的未来需求提前编码（YAGNI）——未来需求是猜的，当前需求是真的
- 功能分层：核心路径 → 错误处理 → 边界优化 → 体验打磨
- 重构也是渐进的：先提取、再优化、最后删旧代码——不要一步到位

### 对结果负责

- 提交了代码 ≠ 完成了工作，任务不是结果，能用、好用才是结果
- 用户打不开、点不动、没反应 \= 工作不合格
- 每一步完成后必须站在用户角度实测，确认"能用且好用"才算完成

## 二、执行纪律

> 本纪律为项目执行的硬底线（不可违反），内化自团队共享工作纪律。任何 AI 助手、开发者、会话必须遵守。

- **熟悉工作原则，并能够灵活运用**
- **搞清楚问题本身和其背景，明确问题和其真实用意再动手，不明白就问**
- **要在一手信息基础上解决问题，而不是靠经验、二手文档解决问题**
- **任务完成先自检，不确定的结果必验证**
- **操作必有记录，凡事有章可循**
- **犯错必记录，知错必改**
- **成功的经验、可复用的能力必沉淀**
- **不越权执行操作**
- **不擅自下载、安装软件**
  - 尽可能复用本地环境中已有的技能、软件、依赖
  - 没有允许绝对不允许擅自下载、安装软件
  - 安装 C 盘需要经过允许，否则不允许随意往 C 盘写东西

## 三、技术开发原则

> 技术工程观（内化自团队共享技术开发原则）。与「工作原则」互补：前者管怎么想，本节管怎么写代码、搭系统。

- **资源节约、更轻更快更强**：按需加载、按需调用，不预支不必要的资源（性能与体积优先）。
- **高内聚、低耦合、模块化、组件化、原子化，模块接口清晰**。
- **重结构、轻代码，极简代码，避免冗余**（与 SOUL「代码铁律」、工作原则「分步行动 / YAGNI」同源）。
- **重体验，用户思维，善于站在用户角度考虑问题**（与工作原则「对结果负责」同源）。
- **真理源自一处，全局引用，绝不硬编码**（与 AGENTS「SSOT 纪律」同源）。
- **文件规范有序，信息同步更新**（与执行纪律「操作必有记录」、AGENTS「信息及时」同源）。
- **退化安全，修改可回溯**：任何优化 / 降级必有 fallback，改动可回退。
