# 项目工作原则

> 本文档是项目所有工作的最高准则。任何 AI 助手、任何开发者、任何会话，必须遵守以下原则。 违反其中任何一条，都视为工作不合格。
>
> \*\*每次会话必须同时读取 \*\*`产品需求说明.txt`，了解产品基本情况，避免重复解释。

---

## 一、工作原则

### 实事求是

- 充分调研，以事实为基础
- **不瞎蒙，不猜测**——不确定的事情必须验证后再说
- 读源码、读文档、读实际文件，不准凭空推断

### 第一性原理

- 思考问题的本质，抓关键，不被表面现象迷惑
- 从用户角度出发，问"用户到底要什么"，而非"技术上能做什么"

### 最佳实践

- 务必结合成功实践，绝不闭门造车
- 参考成熟开源项目的优秀架构

### 策略大于行动

- **最佳策略远胜于反复琢磨细节**
- 先想清楚整体方案，再动手。方向对了，细节才会对
- 想透再执行，不急着写代码

### 结构性看问题

- 结构性思维才能换来极简操作
- 先看全局依赖关系，再动局部代码

### 消除理解偏差，取得共识再行动

- 任何改动前必须与用户沟通确认
- **绝不允许"用户一说完就噼里啪啦敲代码"**
- 动手前必须复述理解，确认与用户意图一致——**理解偏差是最大的浪费**
- 术语和概念必须精确对齐——"直接对话"和"委托执行"是完全不同的模式，不能混用
- 遇到模糊指令，先问清含义再行动，宁可多问一句，不可猜错一步
- 读代码时以实际行为为准，不以注释或文档为准——**代码是唯一的真相**
- 修改前先确认影响范围：改了这个文件，还有哪些文件会受影响？逐个检查

### 分步行动，渐进实现

- 不要一次性完成所有工作
- 每阶段完成后停下来，确认无误再继续
- 每步都要复查验证
- 先跑通最小可用版本，再逐步增强——**能跑 &gt; 完美**
- 每一步必须可独立验证——能跑、能用、能测
- 不为假设的未来需求提前编码（YAGNI）——未来需求是猜的，当前需求是真的
- 功能分层：核心路径 → 错误处理 → 边界优化 → 体验打磨
- 重构也是渐进的：先提取、再优化、最后删旧代码——不要一步到位

### 对结果负责

- 提交了代码 ≠ 完成了工作，任务不是结果，能用、好用才是结果
- 用户打不开、点不动、没反应 = 工作不合格
- 每一步完成后必须站在用户角度实测，确认"能用且好用"才算完成

## 二、执行纪律

### 先想后做

- 搞清楚指令、搞清楚问题、搞清楚影响范围再动手
- 不理解的地方必须问，不准猜

### 自检与验证

- 每改一处，立刻复查语法、逻辑、交叉引用
- 修改完成后必须站在用户角度体验结果
- 改了前端就刷新页面实测，不要假设"应该没问题"
- **过度工程自检**：每次提交前问自己——这段代码有没有违反决策阶梯？有没有更高级的阶梯可以站？能删的删，能短的短

### 经验必须沉淀

- 每次被批评后，必须把教训沉淀到记忆和经验里
- **同样的错误不犯第二次**
- **好的执行经验必须沉淀为文档，以备参考**

### 不擅自下载、安装软件

- 安装任何软件、依赖首先要查看我的电脑中是否已经存在
- 没有允许绝对不允许擅自下载、安装软件
- 同意下载后，在软件允许范围内，尽量安排到其他硬盘，而不是C盘
- **同样的错误不犯第二次**
- **好的执行经验必须沉淀为文档，以备参考**

### 多层验证

- 函数级：输入输出是否符合预期？边界条件是否处理？
- 集成级：组件间数据流是否正确？事件是否正确触发和接收？
- 用户级：站在用户角度，点得动、看得清、反馈及时？
- 异常级：网络断开、文件不存在、数据为空——是否都有合理降级？
- **改了后端必须验证前端，改了前端必须验证后端**——前后端是同一个系统的两面

### 信息更新要及时

- 最新信息通过wiki的方式或者互链的方式保持同步
- 陈旧无用的信息及时处理
- 及时保留修改记录，及时git
